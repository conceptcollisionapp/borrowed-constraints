# Build Decisions

# DECISIONS

## What was built
A React application named "Borrowed Constraints" that allows users to select a domain (from seeded options or by typing a custom one), view "constraints" (assumptions) within that domain, reflect on these constraints, and "challenge" them by adding them to an "Idea Board". The board displays the challenged assumption, the user's reflection, and mock "recombinations" (potential ways to challenge the constraint) generated by a simulated AI.

## Key Technical Decisions

*   **Core State Management:** The application utilizes React's `useState` hooks to manage primary application states including selected domain, custom domain input, the master list of `constraints`, the `challenged` entries on the idea board, the `flipped` state of individual cards, and user `reflections`.
*   **Data Model Denormalization:** Challenged entries are created by denormalizing essential constraint data (`assumption`, `borrowedFrom`, `provocation`, `crackHint`, `reflection`) directly onto the `challenged` array entry. This design ensures that each board entry is self-contained and will display correctly regardless of changes to the original source constraint or if the user navigates away from the domain.
*   **Unique Identifiers:**
    *   Seed constraints are assigned explicit, stable string IDs (e.g., `'c1'`, `'c2'`).
    *   Custom domain IDs are prefixed with `custom-` (e.g., `custom-energy-storage`) to prevent collision with seeded domain IDs when a user types a name that might otherwise match a system-defined slug.
    *   Challenge entries are assigned unique `challengeId` values using `Date.now().toString(36) + Math.random().toString(36).slice(2)` to prevent collisions, even during rapid user interactions.
    *   Ephemeral states like `flipped` card status and `reflections` are keyed by a composite string `${currentDomainId}:${constraint.id}`. This ensures these states are unique per constraint *within its specific domain*, resolving bugs where flipping a card in one domain would inadvertently flip a card with the same `constraintId` in another domain.
*   **User Interface & Accessibility (A11y):**
    *   The card "flip" interaction is correctly implemented using a native `<button>` element with the `aria-expanded` attribute, which screen readers use to announce the state of expandable content.
    *   **Focus Management:** A callback ref (`ref={node => { if (node) node.focus() }}`) is applied directly to each reflection `textarea`. This mechanism ensures that the textarea receives focus immediately upon mount or re-mount (when a card is opened or re-opened), providing a seamless experience for keyboard and screen reader users. `autoFocus` is retained for belt-and-suspenders, though the callback ref is the primary mechanism.
    *   An `aria-label="Your reflection on this assumption"` is added to the `textarea` to provide a clear, programmatic name for screen reader users, addressing concerns about relying solely on `placeholder` text for accessibility.
    *   Domain buttons dynamically update their labels using a global regex (`replace(/-/g, ' ')`) to correctly handle multiple hyphens in domain IDs.
*   **AI Mocking:** A `setTimeout` simulates the asynchronous nature of a real AI API call for generating recombinations. Status indicators (`idle`, `loading`, `done`) are managed per challenge entry.
    *   **Staleness Guard Removal:** An initially implemented stale-response guard for AI calls was removed. This decision was based on the clarification that `reflection` values stored on `challenged` board entries are immutable after their initial creation, eliminating the possibility of a staleness race condition where a reflection could be edited *after* an AI call was dispatched but *before* its response returned. The code now explicitly comments on this immutability.
